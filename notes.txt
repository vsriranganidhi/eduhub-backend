Authentication System

1. Registering super admin by running seed
    - Every user should be associated with an institution. 
    - So, first we create a master institution with a join code which should not be used by any other user. 
    - Then we create the super admin user with first name, last name, email, password and institution id.
    - User role is set to SUPER_ADMIN.

2. Onboarding institutions and Registering college admins
  - Route: POST /institutions
    - Only super admins are allowed to onboard institutions.
    - Super admin sends institution details (name, branch) and college admin details (first name, last name, email) to the endpoint.
    - Join code for the institution and a temporary password for the college admin are generated. resetPassword is set to true.
    - The endpoint creates the institution and college admin user in a single transaction.
    - After successful creation, the college admin is sent an email with the join code, password and login url.
    - If sending email fails and creation of entries in database are successful, then the endpoint throws an error.

3. First time logging in of college admins
  - Route: POST /auth/login
    - college admin sends email and password to the endpoint.
    - Since resetPassword is set to true instead of directly sending a jwt token resetPassword=true is sent in response. Then the college admin should be redirected to a page to reset their password.

4. Forced password reset
  - Route: POST /auth/reset-password
    - college admin sends old password, new password and confirm password to the endpoint.
    - All passwords are stored in passwordHistory table so that they should not be repeated.
    - The new password should not be the same as any of the passwords used before.
    - The endpoint updates the password and sets resetPassword to false.

5. Logging in after password reset
  - Route: POST /auth/login
    - college admin sends email and password to the endpoint.
    - Since resetPassword is set to false a jwt token is sent in response.

6. Inviting teachers via email
  - Route: POST /auth/invite-teacher
    - Only college admins are allowed to invite teachers.
    - College admin sends email id of the teachers to the endpoint and institution id is extracted from the request context.
    - We generate a unique invitation token and store it in the invitation table with an expiration date (7 days), role as TEACHER and also isUsed flag as false initially with the email id of the teacher.
    - We send an email to the teacher with the registration link which has the invitation token and join code as a query parameter.

7. Registering teachers via invitation link
  - Route: POST /auth/register
    - Teacher clicks on the registration link and is redirected to a registration page. Join code, invitation token and role are passed as query parameters.
    - Teacher enters their details (first name, last name) and a password and hits the endpoint.
    - Validation is done using join code and invitation token to check if teacher is registering in institution they are supposed to. Invalid or expired invitation tokens should throw an error.
    - After registering the invitation token is marked as used.

8. Resending invitation to teachers if invitation is expired
  - Route: POST /auth/resend-invitation
    - When invitation is expired, the teacher sees an error in frontend and contacts the college admin.
    - A new invitation is generated and sent to the teacher.
    - Here college admin updates the existing invitation with a new token and expiration date.

9. Registering students
  - Route: POST /auth/register
    - Students are shared the join code by the college admin.
    - Students enter their details (email, first name, last name, registration number), join code and a password and hit the endpoint.
    - Unique registration numbers inside an institution.

10. Logging in of teachers and students
  - Route: POST /auth/login
    - Teachers and students send email and password to the endpoint.
    - A jwt token is sent in response.
